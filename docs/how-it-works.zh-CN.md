# 工作原理

*[English](how-it-works.md) | 中文*

小车在浅色地面上沿黑线行驶。它没有单片机，整个控制闭环就是每一侧的一个光敏传感器、一个
比较器和一个电机。

## 1. 检测线路

车头每一侧都有一个朝下的**光敏电阻（LDR）**对着地面。黑线反射的光少，白色地面反射得多，
所以光敏电阻在线上和在白底上的阻值不一样。

光敏电阻和一个固定电阻组成**分压电路**，把"光的强弱"变成一个电压：

```
+3V ──[ 光敏电阻 ]──┬── 传感器电压  → 比较器
                  [ R ]
                   │
                  GND
```

当传感器移到黑线上方时，照到光敏电阻的光变少，阻值升高，传感器电压跟着变。比较器检测的
就是这个变化。

## 2. 判断：比较器

传感器电压接到 **LM393 比较器**的**反相（−）输入端**，**同相（+）输入端**则是一个由**微调
电位器**设定的参考电压。比较器只判断一件事：传感器比这个阈值更亮还是更暗，然后给出一个
干净的开/关信号。传感器压在线上时更暗、电压更低，所以输出翻到高电平。这个微调电位器是用来
根据光照和具体线路调阈值的。

LM393 是**双**比较器，一片芯片就能管两侧。它做的事情和"读一个传感器、再执行一句 `if`"是
一样的，只不过它在硬件里瞬间完成，不用任何代码。

## 3. 执行：电机

LM393 的输出是**集电极开路（open-collector）**的，只能把输出拉低，所以需要一个到 +3V 的
**上拉电阻**来确定它的高电平。每一路比较器的输出再经一只小**NPN 三极管**去驱动电机，因为
比较器本身带不动电机。两路是**交叉接线**的：每个传感器对应的比较器去驱动*对侧*的电机，正是
这样才让闭环能自动纠偏（见下一节）。电机两端的**续流二极管**保护三极管，挡住电机断电瞬间的
电压尖峰；**指示灯 LED** 用来显示比较器的输出，该侧传感器离开线时它会亮。

## 4. 闭环

两个传感器靠得很近，装在车头。小车对中时，两个都在线上、两个电机都转，所以小车直行。一旦
偏了：

| 情况 | 传感器 | 结果 |
|---|---|---|
| 对中 | 两个都在线上 | 两电机都转 → 直行 |
| 向右偏，线只在左传感器下 | 左在、右离开 | 右传感器离开→切断交叉接线的**左**电机 → 小车向**左**回正 |
| 向左偏，线只在右传感器下 | 右在、左离开 | 左传感器离开→切断交叉接线的**右**电机 → 小车向**右**回正 |

放慢小车要转向那一侧的车轮，就把它带回线上，误差就这样不断被修正。这就是 **bang-bang
控制**：每个电机要么全开、要么全关，没有中间状态。它是最简单的一种闭环，也就是比较器在做的
全部事情。

## 5. 从硬件到仿真

因为真正的判断发生在模拟电路里，本仓库的仿真用软件把同一个闭环重做了一遍，好让你能直接
看到它跑。它是对控制原理的软件建模，不是什么车上固件，因为车上根本没有固件。

模型用 C++ 写（`src/line_follower.h` 和 `src/line_follower.cpp`）。两个虚拟传感器读一条
画出来的线，开/关规则给出两个轮子的速度，再用一个差速模型把车开起来。仿真程序把轨迹记到
一份 CSV 日志里，再由一个 Python 脚本（`render.py`，用 Pillow）逐帧把赛道和小车画进演示
动图。

仿真里这条规则写成"放慢仍在线上那一侧的车轮"。对两个传感器来说，这和电路板的交叉接线是
同一个动作（切断离线那个传感器对侧的电机），只是换成从车轮这一侧来说而已。

有一点只在仿真里有：小车要是在急弯上短暂丢了线，会按上一次的方向继续拐，直到重新找到线。
这只是个软件上的小处理，真实的模拟电路板没有这种记忆。

还有一个无界面的测试（`tests/test.cpp`），跑的是同一套控制和运动代码，只是不渲染。它检查
小车在三条赛道上都贴着线、偏差只有几个像素，并且把转向方向用世界坐标固定下来，这样镜像翻转
的模型会让测试失败。

## 6. 浏览器播放器

仓库里还有一个跑在浏览器里的轻量播放器（`web/`）。它**不**重新跑仿真。C++ 引擎把每条赛道的
轨迹导出成 JSON（`make web` 会写出 `web/data/oval.json`、`wavy.json` 和 `figure8.json`），
播放器把这份 JSON 读进来，在 `<canvas>` 上动画播放。JSON 是仿真程序自己写的：输出路径以
`.json` 结尾时，它写出赛道折线加上每帧轨迹 `[{x, y, theta, sLx, sLy, sRx, sRy, onL, onR}]`；
其它路径则保留 Python 渲染脚本读的 CSV。两种格式都出自同一个仿真步骤。

也就是说，所有实质性的计算都发生在 C++ 引擎里。播放器里没有传感器模型、没有比较器、没有
控制器、也没有运动学：它画出预先算好的赛道线，把两个存好的位姿做插值让运动平滑，再把两个
预先算好的传感器圆点连同它们存好的开/关状态画上去。它在浏览器里扮演的角色，和 `render.py`
之于 GIF 是一样的。
